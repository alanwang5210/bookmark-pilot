# Bookmark Pilot 失效链接扫描与清理设计

## 背景

Bookmark Pilot 当前已经提供：

- 书签与标签页统一搜索
- AI 标签增强
- 索引重建与 AI 标签清理

但还没有“失效链接维护”能力。用户希望新增一个安全的清理流程，用来发现已经失效的书签和当前标签页，并在人工确认后再批量删除或关闭。

## 目标

- 支持扫描书签和当前打开标签页中的失效链接。
- 先展示结果，再由用户确认是否批量清理。
- 清理动作区分书签与标签页：
  - 书签：删除
  - 标签页：关闭
- 将误删风险降到最低。

## 非目标

- 不做自动定时扫描。
- 不在发现失效后自动删除。
- 不把所有非 `2xx` 状态都算作失效。
- 不新增复杂的后台任务调度系统。

## 判定规则

本功能只把以下情况判定为“失效”：

- HTTP `404`
- HTTP `410`
- DNS 解析失败
- 连接失败

以下情况默认不判定为失效：

- `401`
- `403`
- `429`
- `5xx`
- 超时
- 重定向后最终可访问

这样做的原因是尽量避免把临时故障、鉴权限制或站点防护误判为失效。

## 用户体验

### 入口位置

在设置页的“索引与隐私”区域新增一组维护操作：

- 扫描失效链接

扫描完成后，如果存在结果，则在该区域下方显示待处理列表和确认操作。

### 流程

1. 用户点击“扫描失效链接”
2. 扩展开始检测当前索引中的书签和当前打开的标签页
3. 页面显示扫描进度
4. 扫描结束后展示结果列表：
   - 类型（书签 / 标签页）
   - 标题
   - URL
   - 失效原因
5. 用户勾选要处理的项
6. 用户点击“批量清理”
7. 扩展执行：
   - 删除选中的书签
   - 关闭选中的标签页
8. 清理完成后刷新索引与状态

### 安全性

必须采用“先扫描、再确认、后执行”的三段式流程。

任何情况下都不能在扫描完成后自动删除书签或关闭标签页。

## 技术设计

### 数据来源

扫描对象来自现有索引数据：

- 书签：`SearchIndex.bookmarks`
- 标签页：`SearchIndex.tabs`

只检测带 `http` 或 `https` URL 的项。

### 检测方式

后台新增链接检查逻辑，对每个 URL 发起轻量请求。

建议策略：

- 使用 `fetch`
- 默认跟随重定向
- 只使用 `GET`
- 捕获网络错误并映射为可读原因

判定逻辑：

- 最终响应码为 `404` 或 `410` => 失效
- `fetch` 抛出网络连接类错误 => 失效
- 其它情况 => 保守处理为“未命中失效”

### 后台消息

建议新增消息类型：

- `SCAN_DEAD_LINKS`
- `GET_DEAD_LINK_SCAN_PROGRESS`
- `CLEAN_DEAD_LINKS`

其中：

- `SCAN_DEAD_LINKS`：启动扫描并返回初始状态
- `GET_DEAD_LINK_SCAN_PROGRESS`：轮询扫描进度和结果
- `CLEAN_DEAD_LINKS`：接收确认后的项目列表并执行删除/关闭

### 结果结构

建议扫描结果结构如下：

```json
{
  "id": "bookmark:123",
  "type": "bookmark",
  "entityId": "123",
  "title": "Example",
  "url": "https://example.com",
  "reason": "404"
}
```
```

对于标签页：

```json
{
  "id": "tab:456",
  "type": "tab",
  "entityId": "456",
  "tabId": 456,
  "title": "Broken Tab",
  "url": "https://broken.example.com",
  "reason": "DNS failure"
}
```

### 清理执行

确认后：

- `bookmark` 项使用 `chrome.bookmarks.remove`
- `tab` 项使用 `chrome.tabs.remove`

执行完成后需要：

- 刷新书签索引
- 刷新标签页索引
- 更新设置页状态和反馈信息

### 结果展示

设置页新增一块扫描结果区域，包含：

- 扫描进度
- 命中数量摘要
- 可勾选列表
- 批量清理按钮

如果没有命中失效项，则显示明确反馈，例如：

- 未发现失效书签或标签页

## 错误处理

- 单个 URL 检测失败不应中断整个扫描任务
- 清理单项失败不应阻断其它项继续执行
- 页面需要给出最终成功/失败数量摘要

## 风险与边界

- 某些网站可能对程序化请求有限制，存在少量误判风险
- 当前标签页在扫描结束前可能已经被用户关闭
- 某些书签可能在确认前已被用户手动删除

因此，执行清理时必须按项目分别容错。

## 验证要求

实现完成前至少验证：

- 能扫描书签和标签页
- `404` 与 `410` 会被识别为失效
- DNS/连接失败会被识别为失效
- `403`、`500` 不会被直接清理
- 扫描只展示结果，不自动删除
- 确认后书签会被删除、标签页会被关闭
- 执行后索引状态能刷新
